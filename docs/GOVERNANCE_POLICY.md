# Governance Policy

## Purpose
This document is the canonical source of enforceable governance policy for this repository.

The goal is to:
- prevent policy drift across documents,
- keep expectations clear for contributors,
- enforce integrity in high-stakes domains.

## Document Hierarchy
When governance text overlaps, interpret documents in this order:
1. `docs/CONSTITUTION.md` (foundational principles)
2. `docs/GOVERNANCE_POLICY.md` (enforceable policy and process)
3. `CONTRIBUTING.md` (contributor workflow)
4. `README.md` (project orientation and navigation)

## Foundational Integrity Standard
At all times:
- Never lie.
- Never cheat.
- Never steal.

Non-complicity is mandatory:
- Do not request, encourage, or assist these violations.
- Do not conceal known violations.
- Escalate suspected violations through repository channels.

## Critical Review and Feedback Protocol
All aspects of this project are open for critical review and feedback.

Approved channels:
1. GitHub issues
2. GitHub discussions
3. Pull request reviews

Submission format for critical review:
1. One instruction per bullet point
2. Maximum 20 words per bullet point
3. Use Mermaid graphs when useful to clarify structural changes
4. Do not include sensitive information; all feedback is public

## Safety-Critical Handling
If an integrity concern affects safety-critical work:
1. Raise the concern immediately in project channels.
2. Pause affected work until review by maintainers.
3. Resume only after a documented disposition is posted.

## Enforcement
Violations of integrity or non-complicity expectations may result in:
1. Request for correction
2. Rejection or rollback of proposed changes
3. Suspension of contribution privileges
4. Escalation to maintainers for additional action

## Policy Change Management
All governance policy updates must:
1. Be proposed through a pull request that references an issue.
2. Include a concise rationale and expected impact.
3. Update `docs/GOVERNANCE_CHANGELOG.md`.
